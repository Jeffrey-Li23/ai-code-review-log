## 审查报告

### 文件 1：`.github/workflows/main-remote.yml`

#### 变更概要
原有的环境变量定义：
```yaml
GH_TOKEN: ${{ GH_TOKEN }}
DEEPSEEK_API_KEY: ${{ DEEPSEEK_API_KEY }}
MAIL_SENDER: ${{ MAIL_SENDER }}
MAIL_AUTH_CODE: ${{ MAIL_AUTH_CODE }}
MAIL_RECIPIENT: ${{ MAIL_RECIPIENT }}
```
修改为：
```yaml
GH_TOKEN: ${{ secrets.GH_TOKEN }}
DEEPSEEK_API_KEY: ${{ secrets.DEEPSEEK_API_KEY }}
MAIL_SENDER: ${{ secrets.MAIL_SENDER }}
MAIL_AUTH_CODE: ${{ secrets.MAIL_AUTH_CODE }}
MAIL_RECIPIENT: ${{ secrets.MAIL_RECIPIENT }}
```

#### 发现的问题

| 问题 | 严重等级 | 说明 |
|------|----------|------|
| 使用 `${{ VAR }}` 而不是 `${{ secrets.VAR }}` 引用机密 | **严重** | 在 GitHub Actions 中，`${{ GH_TOKEN }}` 会直接引用上下文字段的变量（如 `github.token`），而 `${{ secrets.GH_TOKEN }}` 才是正确的机密引用方式。原写法会导致以下后果：<br> - 若该变量未定义，则运行时会被替换为空字符串；<br> - 若该变量意外被打印或记录，可能导致秘密泄露；<br> - 无法利用 GitHub Secrets 的自动遮蔽和审计功能。 |
| 缺少对 Secret 存在性的校验或 fallback | **警告** | 修正后正确使用了 `secrets` 上下文，但工作流中没有额外对缺失 Secret 进行显式检查（例如通过 `if: secrets.GH_TOKEN != ''` 跳过关键步骤），不过这在大多数项目中是可接受的，因为若有 Secret 缺失，`java -jar` 步骤可能会因缺少令牌而失败，并暴露一些错误信息。建议添加一个前置步骤用于验证必需 Secret 是否存在。 |

#### 修改建议

1. ✅ 本次修改已正确修正为 `${{ secrets.GH_TOKEN }}` 等，符合 GitHub Actions 安全最佳实践。
2. **建议**（可选）添加一个快速检查步骤，避免因 Secret 缺失导致后续步骤失败并暴露敏感信息：

```yaml
- name: Validate required secrets
  run: |
    if [ -z "${{ secrets.GH_TOKEN }}" ]; then
      echo "GH_TOKEN is missing"
      exit 1
    fi
    # 同样检查其他必需 secrets
```

#### 其他注意点

- 该工作流中使用了 `java -jar ./libs/ai-code-review-1.0.jar` 命令，暗示 jar 包需要读取这些环境变量。确保 Java 代码已使用安全方式（如 `System.getenv("GH_TOKEN")`）获取环境变量，且不会在日志或异常中输出原始值。
